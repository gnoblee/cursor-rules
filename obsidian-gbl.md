# Obsidian_GBL Project Rule (Cursor)

## 0) 기본 원칙
- 항상 한국어.
- 작업 시작 전: (1) 내가 이해한 요청 1줄 (2) 필요한 추가 질문 최대 3개 (3) 바로 실행 가능한 다음 행동 제시.
- 대량 변경(다수 파일 이동/이름변경/일괄 수정)은 **반드시 사전 확인** 후 진행.
- 파일을 수정할 때는 **frontmatter(YAML)를 보존**하고, 사용자가 요청하지 않으면 태그/상태값을 임의로 바꾸지 않는다.

## 1) 절대 금지/주의
- `90_도구/98_시크릿/` 관련 내용은 열람/요약/복사/추측/재출력 금지(민감정보 가능성).
- 첨부/이미지/Excalidraw는 기본적으로 분석 대상에서 제외(필요 시 사용자에게 요청).

## 2) 이 볼트의 폴더 의미(경로 기준)
- `00_인풋박스/` : 임시 수집. 정리 요청이 오면 “분류→이동→링크/태그 최소화” 중심.
- `10_자료모음/` : 레퍼런스 보관. 원문 훼손 최소.
- `20_생각노트/` : 아이디어/연결 강화. 링크(내부링크) 우선.
- `30_프로젝트/` : 실제 작업. `32_대기목록 → 33_진행중 → 39_완료/연도` 흐름 존중.
- `40_퍼블리쉬/` : 발행 파이프라인. 초안은 `41_초안`, 발행 후 `42_발행완료/플랫폼`.
- `70_사람목록/` : CRM. 개인/고객 정보는 노출 최소, 요약 시 식별정보 마스킹.
- `90_도구/` : 템플릿/설정/스니펫. 시스템 변경은 사전 확인 필수.

## 3) 노트 작성/수정 규칙(옵시디언 친화)
- 제목(H1)은 파일명과 일치시키는 것을 기본으로 제안(강제 X).
- 내부 링크는 `[[노트명]]` 우선. 폴더 이동이 필요하면 “링크 깨짐 여부”를 먼저 점검하도록 안내.
- 체크리스트/작업은 가능한 한 “다음 행동(Next Action)” 단위로 쪼개서 제안.
- 긴 글은 `요약 → 본문 → 체크리스트/다음 단계` 구조를 기본으로 한다.

## 4) 자주 하는 작업별 처리 방식
### (A) 인풋박스 정리 요청
- 1) 어떤 성격인지 분류(자료/생각/프로젝트/발행) 제안
- 2) 이동 경로 추천
- 3) 필요한 최소 링크만 추가(과도한 태그 금지)

### (B) 프로젝트 문서화
- 기존 체계가 있으면 그에 맞춘다.

### (C) 발행(퍼블리쉬) 작업
- 초안 생성/개선은 `40_퍼블리쉬/41_초안/`에서 진행.
- 발행 완료 시 `42_발행완료/<플랫폼>/`로 이동 제안 + 발행 URL/날짜 기록(요청 시).

## 5) 출력 포맷(기본)
- “추천 행동 3개”를 항상 마지막에 제공:
  1) 지금 바로 할 일
  2) 다음 단계
  3) 확인 질문(있다면)
